### 2. Testes unitários criados
**Arquivo:** `backend/src/test/java/com/gastos/service/CategoryServiceTest.java`

Usa `@ExtendWith(MockitoExtension.class)` com `@Mock CategoryRepository` e `@InjectMocks CategoryService`.

Cenários cobertos (12 testes):
- `findAll(null)` → retorna todas as categorias ativas
- `findAll(EXPENSE)` → retorna EXPENSE + BOTH
- `findAll(INCOME)` → retorna INCOME + BOTH
- `findById` → encontrada → retorna DTO com todos os campos
- `findById` → não encontrada → `EntityNotFoundException`
- `create` → sucesso → salva e retorna DTO
- `create` → nome duplicado → `IllegalArgumentException`
- `update` → sucesso → atualiza nome e cor
- `update` → versão errada → `ObjectOptimisticLockingFailureException`
- `update` → não encontrada → `EntityNotFoundException`
- `delete` → sucesso → seta `inactive = true`
- `delete` → não encontrada → `EntityNotFoundException`

### 3. Testes de integração criados

Cenários cobertos (11 testes, @Order 1–11):

2. GET com token → 200

6. POST nome duplicado → 409
7. PUT válido → 200 com nome/cor atualizados
8. PUT ID inexistente → 404
9. DELETE válido → 204
10. DELETE ID inexistente → 404
11. GET após DELETE → categoria não aparece na lista (soft delete verificado)

## Pendências / Riscos identificados

- `bearerToken` e `createdCategoryId` são `static` no teste de integração para compartilhar
  estado entre métodos `@Order`. Se testes rodarem em paralelo ou em ordem diferente,
  podem falhar — uso de `@TestMethodOrder(MethodOrderer.OrderAnnotation.class)` mitiga isso.
- Caso o H2 não honre a constraint UNIQUE em `name` da entidade `Category`, o cenário de
  "nome duplicado" passará pela validação no service (via `existsByNameIgnoreCaseAndInactiveFalse`)
  antes de chegar ao banco — sem impacto funcional, mas merece observação.
- O cenário de `update` com versão errada testa a verificação **manual** feita no service
  (`if (!category.getVersion().equals(request.version()))`), não o mecanismo nativo do Hibernate
  — válido e correto dado o design do service.
